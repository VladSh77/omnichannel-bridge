# MEMORY — Fajna / omnichannel_bridge

> Архітектурні рішення, інциденти, критичні застереження для розробника.

---

## ADR-001: Локальний LLM (Ollama) як основний бекенд

**Рішення:** Ollama з відкритими вагами, OpenAI — опційно.
**Чому:** Без ліцензій App Store, без обов'язкових платних API, GDPR (дані не йдуть у хмару), інференс на власному сервері.
**Наслідок:** Ollama HTTP API (`http://127.0.0.1:11434`) — Odoo повинен мати мережевий доступ. Якщо Ollama ліг — fallback без LLM (шаблонна відповідь), не мовчання.

---

## ADR-002: Факти лише з Odoo ORM, не з голови моделі

**Рішення:** `omni_knowledge.py` збирає дані через ORM (`product.template`, `event`, `res.partner`) і передає в промпт як структурований контекст.
**Чому:** Преміум-домен (діти, гроші, безпека) — LLM не може вигадувати ціни, місця, умови.
**Наслідок:** При зміні схеми кастомних модулів campscout — оновлювати `omni_knowledge.py`.

---

## ADR-003: Паралельне існування з SendPulse

**Рішення:** omnichannel_bridge — окреме підключення, не міграція SendPulse.
**Чому:** Мінімізація ризику — старий потік залишається живим поки новий не протестований.
**Наслідок:** Два вебхуки в Odoo можуть конфліктувати якщо один канал підключений до обох.

---

## ADR-004: Гібрид бот + один менеджер (не черга)

**Рішення:** Один відповідальний менеджер на діалог + бот поза робочими годинами.
**Чому:** Преміум-позиціонування — клієнт не хоче відчувати «колл-центр».
**Наслідок:** ⚠️ Гонка бот + менеджер — якщо менеджер онлайн і пише, бот має мовчати. Kill switch на рівні каналу або статусу менеджера.

---

## ADR-005: LGPL-3 ліцензія модуля

**Рішення:** LGPL-3 (стандарт Odoo Community).
**Чому:** Сумісність з Odoo LGPL-3 ліцензією.
**Наслідок:** Публічне репо — не розкривати клієнтські дані, токени, IP в коді.

## ⚠️ КРИТИЧНІ ПРОБЛЕМИ ПРОДАКШЕНУ

### 1. Ідемпотентність вебхуків
Meta і Telegram можуть повторно доставити подію. Без дедублікації → дублі в Discuss + плутанина клієнта.
**Статус:** перевірити `omni_bridge.py` — чи є перевірка `message_id` перед створенням.

### 2. LLM таймаут у HTTP-запиті
Ollama може відповідати 10–30 сек. Odoo HTTP worker + Nginx мають ліміти (~60 сек).
Meta очікує відповідь вебхука за 20 сек — інакше повторна доставка.
**Рішення:** або асинхронна черга (ir.actions.server + cron), або streaming + швидка відповідь 200 OK Meta до того як LLM завершить.

### 3. Гонка бот + менеджер
Якщо менеджер пише і бот паралельно — два повідомлення підряд = втрата довіри.
**Рішення:** kill switch на рівні каналу (`omni_bridge`: `bot_active` поле на `mail.channel`).

### 4. Мінімізація ПІБ дітей у промпті
GDPR/RODO + діти (особлива категорія). Ім'я дитини, вік, медичні деталі — не мають іти в LLM без явної необхідності.
**Рішення:** фільтр в `omni_ai.py` — що саме передається в контекст.

### 5. Логи без ПІБ
`omni_memory.py` і логи Odoo не повинні писати ПІБ клієнтів у відкритий текст.
**Рішення:** маскування в debug logs.

### 6. Kill switch
Вимкнути бота за 1 хвилину без деплою — поле в `res.config.settings` або на каналі.
**Статус:** перевірити чи є в `res_config_settings.py`.

### 7. Staging = ті самі кастомні модулі
Тестування без campscout_management → "на сервері все інакше".
**Рішення:** staging сервер з тими самими addons_path.

### 8. Fallback без LLM
Якщо Ollama недоступний → шаблонна відповідь ("Зараз уточнюємо..."), не мовчання.
**Статус:** перевірити try/except в `omni_ai.py`.

## Інтеграція з campscout кастомними модулями

Схема квиток/подія/місця — читати з `campscout_management` моделей, не з абстрактного Odoo.
Перед оновленням `omni_knowledge.py` — перевіряти актуальні моделі в campscout.eu репо.
