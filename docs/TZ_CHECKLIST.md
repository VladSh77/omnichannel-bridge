### Інтеграція з існуючим Odoo: читати код на сервері, не «вгадувати» схему

**Чому так:** у продакшені вже є **власні кастомні модулі**, яких **немає в базовому (стандартному) Odoo**. Саме вони часто визначають, як рахуються **місця, квитки, реєстрації, знижки** тощо. Модуль `omnichannel_bridge` **має співпрацювати з цим кодом**, а не будуватися лише під «типові» моделі Community/Enterprise — тому схему даних і бізнес-правила знімають **безпосередньо з вашого репозиторію аддонів і БД на сервері**, а не з загальних підручників.

Зв’язок **квиток → подія → зайняті / вільні місця** і будь-які обчислення квот **не визначаються з одного лише ТЗ** чи документації «чистого» Odoo. Реальність інстансу формують:

- **Ваші кастомні модулі** (пріоритет при аналізі) + **сторонні** (OCA, інтеграції) — нові/змінені моделі, `inherit`, `related`, `compute`, обмеження SQL.
- **Переклади в інтерфейсі** (українська/польська) можуть бути неконсистентними або вводити в оману — **надійні технічні імена** лише в **Python/XML** і в ORM.
- «Поганий переклад» не скасовує фактичну логіку в коді; інтеграція мусить орієнтуватися на **реальні поля та методи**, а не на підписи в формі.

**Обов’язкова практика розробки (на staging або production-клоні з тим самим кодом):**

1. Переглянути **встановлені додатки** (`Apps`) і `addons_path` — окремо виділити **ваші кастомні** пакети (не з upstream Odoo).
2. Прочитати **вихідний код** цих модулів і залежностей (`models/*.py`): `_name`, `inherit`, поля, `compute`, домени пошуку, стани (`state`, `active`), скасування броні.
3. За потреби — **Odoo shell** / дебаг: `env['model']._fields`, тестовий `search_count` для однієї події, звірка з UI.
4. Результат зафіксувати в **внутрішній техдок** репозиторію або wiki: таблиця «модель / поле / що означає / як рахується вільне місце / коли рядок блокує місце».
5. Після **оновлення Odoo, оновлення/додавання модуля** — повторити перевірку (регресія інтеграції).

### 2.2 Вхід у діалог на сайті: pre-chat або «розумні» кнопки (узгоджено з продуктом)

**Проблема, яку закриваємо:** у неробочий час (і часто на першому кроці) користувачі бачать лише кнопки без вільного вводу або навпаки — одразу відкритий чат без контексту. Потрібен передбачуваний UX: **або збір контакту + згода, або кнопки, що дають реальну цінність**, і завжди можливість **залишити контакт**.

**Еталон UX (нативний Odoo Website Live Chat / чатбот):** одне повідомлення бота з питанням, під ним **вертикальний список кнопок-відповідей** (quiz), а внизу віджета **завжди видимий рядок вводу** («Скажіть щось…») — користувач може або натиснути варіант, або одразу написати власний текст. Це **цільовий вигляд** для CampScout: не допускати стану, коли замість поля вводу показується лише підказка на кшталт «Виберіть опцію вище» без композера.

**Два допустимі варіанти (обрати один як основний або комбінувати за правилами каналу):**

1. **Pre-chat / стартова форма (аналог SaaS «pre-chat form»)**  
   - Перед повноцінним полем вводу показується короткий блок: ім’я + **хоча б один** з каналів зв’язку (email **або** телефон), опційно — причина звернення.  
   - Поруч — **коротке RODO-пояснення** (контролер, мета, посилання на політику; без вигаданого юридичного тексту в боті).  
   - Після відправки форми відкривається **повний чат** (вільний текст + AI/оператор за правилами каналу).  
   - У Odoo: реалізація через **кроки чатбота Live Chat**, окремий **website snippet** або кастомний крок у `omnichannel_bridge` — технічний вибір фіксується в техдокі після оцінки.

2. **Кнопки як навігація по цінності, не як «глухий коридор»**  
   Якщо на першому екрані використовуються кнопки, кожна гілка має вести до **конкретної користі**, а не лише до загального «привіт»:  
   - **Про компанію CampScout** — 2–4 речення + посилання на затверджені сторінки (про нас, ліцензія/довіра — лише з фактів сайту/Odoo).  
   - **Табори / зміни** — короткий огляд + посилання на каталог/події на сайті; у чаті далі — уточнення віку, дат, бюджету (узгоджено з sales discovery).  
   - **Інші послуги** (якщо є в продуктовій лінійці) — окремі кнопки або підменю, щоб не змішувати з табором без явного вибору.  
   - **Ціни / умови** — відповідь лише з **FACTS з Odoo** + посилання на оферту; без вигаданих цифр.  
   - **Залишити контакт / передзвін / email** — **обов’язкова** гілка або фінальний крок після будь-якої з гілок: ім’я + email або телефон, підтвердження згоди.  
   - **Одночасно з кнопками** має лишатися доступним **повний композер** (як у еталоні Odoo вище); додаткова кнопка «Написати своє питання» — опційна, якщо з коробки вже є вільний ввід.

### 6.1 Гібрид «бот ↔ один менеджер» (ніч, 3 хв, пауза)

**Контекст:** один менеджер, графік нерегламентований — бот закриває **ніч**, **дуже ранок**, паузи та **відсутність відповіді ~3 хв**; коли менеджер активний — бот **не конкурує** за тон спілкування.

- [x] **Ніч / ранок:** опційне вікно `night_bot_*` + затримка черги під час робочих годин менеджера на SLA (`bot_inside_hours_if_manager_quiet`, `omni_autoreply_delay_seconds_for_inbound`) — Meta/Telegram; live chat без зміни.
- [x] **Таймер ~3 хв:** після повідомлення клієнта, якщо за N секунд **немає** відповіді від людини (не OdooBot) у треді — дозволити відповідь бота (черга/cron + перевірка `mail.message`).
- [x] **Пауза бота при менеджері:** якщо менеджер **написав** у треді, запускається manager-session lock (`manager_session_active`) з timeout; у цей період inbound не enqueue-иться в LLM.
- [x] **Миттєве підхоплення:** опція «бот відповідає одразу, менеджер підключається пізніше» — сумісна з паузою після першої репліки менеджера.
- [x] Логування: хто відповів (бот / менеджер), щоб аналізувати якість (додано CRM Analytics метрики `bot_reply_threads` / `human_reply_threads` / `mixed_reply_threads`).

| Тип | Приклади | Логіка відповіді |
|-----|-----------|------------------|
| **Ціна / бюджет** | «Дорого», «не готовий платити стільки» | Підтвердити емпатію → **факти** з Odoo (прайс-лист, складники цінності, розстрочка/етапи оплати — лише якщо є в системі) → запитання на уточнення бюджету / порівняння з альтернативою |
| **Час** | «Пізніше», «після відпустки» | Зафіксувати дату в CRM → легкий наступний крок (нагадування в **дозволеному вікні** каналу або перехід у Telegram) |
| **Довіра** | «Не знаю вашу компанію», «перший раз чую» | Соціальний доказ **лише правдивий** (реальні відгуки/цифри з дозволених джерел); без вигаданих нагород |
| **Потреба в «подумати»** | «Я подумаю» | Уточнити, над чим саме сумнів → одне чітке питання; запропонувати менеджера або коротке резюме переваг **з фактів** |
| **Конкурент** | «У конкурента дешевше» | Не обговорювати конкурента на емоціях; перенос на **вашу пропозицію** (що входить, гарантії, умови з Odoo) |
| **Не я ЛПР** | «Треба порадитись з …» | Запитати роль ЛПР, зручний час дзвінка/чату, **handoff** менеджеру |

**Принципи (кращі практики):**

1. **Спочатку прийняти** («розумію, це важливе рішення»), не сперечатися з першої репліки.
2. **Ізолювати** заперечення: «Якщо вирішимо питання ціни, зручно буде продовжити?»
3. **Питання краще тверджень** — щоб не звучати тиском.
4. У відповідях бота — **лише підтверджені дані** (блок FACTS); якщо аргументу немає в Odoo — чесно сказати і запропонувати людину.
5. **Ескалація:** при юридичних/медичних/страхових питаннях або агресії — одразу **менеджер**, бот не імпровізує.

**Чекліст реалізації в продукті:**

- [x] Детектор **типу заперечення** (ключові фрази + легкий класифікатор): додано rule+score classifier (phrase/token scoring) у `omni.sales.intel`.
- [x] Прив’язка до **готових блоків відповіді**: `OBJECTION_PLAYBOOK` у system prompt + Settings + окремий CRM/UI policy-editor (`omni.objection.policy`).
- [x] Після відпрацювання заперечення — **наступний крок** у воронці (наприклад, мікро-зобов’язання клієнта)
- [x] Лог у партнера: **які заперечення** вже були — рядки `objection:*` в `omni_chat_memory` + внутрішня нота в Discuss.
- [x] Метрики: частота типів заперечень, конверсія після відпрацювання (додано `objection_to_intent_percent` у CRM Analytics).

## 13. Стимул переходу в Telegram (знижка 5% та канал для розсилок)

**Не плутати** з **§ 8.1** — там **внутрішній** канал для менеджера та керівника (операційні сповіщення). Тут — **клієнтський** Telegram (бот, купон, маркетингові повідомлення за згодою).

**Бізнес-ідея:** у **Telegram** немає обмеження **24-годинного вікна** як у Meta/WhatsApp для типового сценарію бота — зручніше **довгі нуртування**, нагадування про акції, **розсилки** (за згодою). **5% знижки** за перехід — сильна мотивація, якщо коректно оформити в обліку.

**Узгоджено для таборів (оновлено):** знижка **лише на продукт «табір»**; клієнт самостійно переходить у публічний Telegram-канал `@campscouting` і бере актуальний код з pinned/latest поста.

**Оцінка (для стейкхолдерів):** рішення **логічне** для вашої моделі (реклама в Meta → швидко «дотягнути» до каналу без вікна). Ризики: **маржа**, зловживання (один клієнт — кілька акаунтів), узгодження з бухгалтерією/договором оферти. Компенсація: **обмеження продуктом** + **одноразовість** купона за правилами.

**Чекліст реалізації:**

- [x] Правила акції в **Odoo**: купон/промо «−5%» валідується через `omni.promo` (активність/дати/код/scope) і застосовується лише до camp lines (та whitelist `product_tmpl_ids`, якщо задано).
- [x] Генерація/видача унікального коду **не використовується**: код публічний у Telegram-каналі `@campscouting` (business rule).
- [x] Умова отримання коду: клієнт відкриває публічний канал `@campscouting` і копіює код (без персонального `opt-in` прапорця).
- [x] Повідомлення в Meta/IG: **прозора оферта** runtime-правилом у `omni.ai` (лише табір, як отримати код у `@campscouting`, як застосувати при реєстрації).
- [x] **Згода на розсилки** в Telegram: runtime opt-in/opt-out команди (`/subscribe`, `/unsubscribe`) з фіксацією в `res.partner`.
- [x] Сегментація для розсилок: wizard `Telegram Broadcast` з `only_opted_in` + виключення recent recipients (frequency cap by days).
- [x] Аналітика: у `CRM Analytics` додано метрики переходів в Telegram (`tg_new_contacts`), використання купона (`coupon_redemptions_count`) і ROMI (revenue vs ad spend).

## 14. Оцінка senior: що ще критично важливо для ТЗ (часто недоговорюють)

### 14.1 Надійність вебхуків і черг

- [x] **Ідемпотентність:** Meta/Telegram можуть **повторно доставити** той самий webhook — потрібен захист від **дублікатів** повідомлень у Discuss (ключ: `update_id` / `message.mid` / hash payload + збереження «вже оброблено»).

- [x] **Право на видалення:** `res.partner.action_omni_right_to_erasure` + окремий SOP з platform limitations (`docs/RIGHT_TO_ERASURE_SOP.md`).

## 17. Mobile-first message policy (mandatory)

Цей блок є **обовʼязковою частиною ТЗ** для всіх клієнтських каналів (Meta/IG, Telegram, WhatsApp, Viber, livechat).

### 17.1 Канальні ліміти та операційні caps

- Орієнтовні platform hard limits (потрібно періодично звіряти з офіційними docs провайдера):
  - Telegram Bot API: до ~4096 символів на повідомлення.
  - WhatsApp Business/Cloud API (text): до ~4096 символів.
  - Meta Messenger/Instagram: різні обмеження за типом повідомлення/API; використовувати консервативні короткі тексти.
- **Операційні caps у нас (строгіші за platform limits):**
  - Meta/Instagram/WhatsApp/Viber: до ~320 символів в одному повідомленні.
  - Site livechat: до ~360 символів.
  - Telegram: до ~500 символів.

### 17.2 UX правила для діалогу

- Один меседж = одна думка.
- Один бот-відповідь = максимум одне питання до клієнта.
- Заборонені “стіни тексту” та багатопунктні маркетингові блоки одним куском.
- Якщо потрібно більше пояснень — розбивати на 2-3 короткі повідомлення.
- У mobile-каналах пріоритет читабельності: короткі абзаци, прості CTA, мінімум прокрутки.

### 17.3 Data-aware flow (не питати те, що вже відомо)

- Перед кожним запитанням перевіряти наявні поля профілю (`user_fullname`, `user_email`, `user_phone` та інші з контексту).
- Якщо поле вже заповнене — пропускати крок і переходити далі.
- Якщо поле порожнє — ставити тільки одне цільове запитання.

### 17.4 Compliance та безпечне кодування

- Тексти мають коректно відображатися в різних месенджерах (UTF-8, без “битих” символів).
- Емодзі дозволені як маркери структури повідомлення (за потреби), але без перевантаження.
- RODO/Privacy посилання давати коротким рядком, без довгих юридичних вставок у кожній відповіді.

## 18. Silence recovery policy (24/7 sales psychology)

Ціль: не втрачати лідів, які написали і “зникли”, та коректно повертати їх у діалог без спаму.

### 18.1 Сигнали мовчання (поведінкові)

- `client_silent_after_bot`: клієнт не відповів після відповіді бота.
- `client_returned_late`: клієнт повернувся після довгої паузи (пізно ввечері/вночі або через кілька днів).
- `handoff_silent`: клієнт просив менеджера, але не залишив контакт / не завершив крок.

### 18.2 Обов’язкові SLA-тригери

- **T+15 хв** після останньої відповіді бота і без клієнтської відповіді:
  - короткий “nudge” (1 речення, 1 CTA), без маркетингових блоків.
- **T+12 год**:
  - другий nudge з варіантами швидкого вибору (ціна/дати/місця/контакт).
- **T+48 год**:
  - фінальний м’який follow-up + опція “пауза/відписка”.

### 18.3 Нічний режим (24/7)

- Бот відповідає вночі так само швидко, але:
  - без агресивних продажних формулювань;
  - з фразою “менеджер підключиться у робочі години”, якщо потрібен handoff.
- Якщо вночі клієнт дає контакт — фіксуємо lead одразу, нотифікуємо менеджера в ранковий слот.

### 18.4 Анти-спам та частотні обмеження

- Не більше 2 follow-up повідомлень за 48 год на один активний діалог.
- Якщо клієнт відповів — лічильник silence-follow-up скидається.
- Якщо клієнт явно відмовився/“неактуально” — follow-up не надсилати.

### 18.5 Контент правилa для silence follow-up

- Максимум одне питання в повідомленні.
- Довжина за mobile caps із §17.
- Тон: емпатичний, без тиску (“зручно/підкажу/можу допомогти”).
- У разі “дорого/подумаю/сам напишу”:
  - не тиснути, дати короткий наступний крок та контрольну точку повернення.

### 18.6 KPI для відділу продажів

- `response_recovery_15m`: частка діалогів, де клієнт відповів після першого nudge.
- `silence_to_contact`: частка мовчазних діалогів, що завершилися контактом.
- `silence_to_handoff`: частка мовчазних діалогів, що перейшли в менеджерський handoff.
- `overfollowup_rate`: частка порушень частотного ліміту (має бути ~0).

Зміст і **макет** діалогу мають відповідати еталону панелі (п. 2–6 нижче); у репозиторії реалізація зафіксована в §20.8.1 (`[x]`) + `docs/DISCUSS_CONVERSATION_CARD_INSTANCE_ACCEPTANCE_2026-04-10.md` для ручного приймання на інстансі.
2. **Верхній блок:** аватар (або плейсхолдер без «камери» Odoo, поки немає `image` у партнера); **ім’я**; рядок **емодзі каналу + підпис провайдера** (напр. ✈️ Telegram); **бейджі** з payload `channel_profile.badges` (мінімум «Активний» де доречно; для Telegram — додатково **Telegram Premium**, якщо `is_premium`).
3. **Контакт:** основний **ідентифікатор у каналі** (username Telegram, WA id, PSID тощо) — **клікабельне посилання**, якщо є стабільний публічний URL (`t.me`, `wa.me`, `m.me` тощо); **мова** (`language_code`, коли є); телефон з **поділеного контакту** Telegram (якщо є); email/телефон з картки партнера Odoo.
4. **Блок профілю каналу** (`channel_profile.section`): заголовок і рядки з identity metadata — для Telegram це відповідає **getChat** (біо; **active usernames** — кожен нік окремим **клікабельним** `t.me`; дата народження за наявності). **Не** показувати в UI сирі дампи Bot API чи зайві внутрішні id «для діагностики».
5. **Клієнт Odoo:** ім’я контакту (акцент як посилання на внутрішню картку); email/телефон; за потреби — попередження **чернетка профілю**; опційно рядок згоди на **розсилку CampScout** (Telegram).
6. **Тред:** назва каналу в Discuss; **зовнішній id** треду; за наявності — підписаний **дубль id користувача в месенджері** (напр. числовий user id Telegram), якщо він відрізняється від рядка зовнішнього id або дає додаткову змістовність.

**Критерій приймання:** макет і логіка вище виконуються для omnichannel-тредів **кожного** підключеного каналу; картка без службових «пояснювальних» абзаців у підвалі панелі. Те саме стосується **діалогу «Картка розмови»** після кліку по стрілці — перевірка на проді за чеклістом у `docs/DISCUSS_CONVERSATION_CARD_INSTANCE_ACCEPTANCE_2026-04-10.md`.
